Azure DevOps Pipeline Architecture

Defined in azure-pipelines.yml at the project root.

Stage 1: Build & Package

Runs for both dev and main branches

  • Setup Node.js 20
  • npm ci with cache
  • npm run build:deploy
  • Copy frontend/build → backend/static
  • Verify index.html exists
  • ZIP backend/ folder
  • Publish artifact
Stage 2: Deploy DEV
AUTOMATIC

Condition: branch = dev

  • Setup Python 3.12
  • Alembic migrate
  • ZIP Deploy
  • Health check
Target: NorthIntelligenceDev
RG: NTG-Analytics-RG-Dev
Stage 3: Deploy PROD
MANUAL APPROVAL

Condition: branch = main

  • Setup Python 3.12
  • Alembic migrate
  • ZIP Deploy
  • Health check
Target: NorthIntelligence
RG: NTG-Analytics-RG-Prod

Azure DevOps Configuration (One-Time Setup)

1. Service Connections

Go to: Project Settings → Service connections → New service connection

Connection Name Type Scope
north-intelligence-azure-dev Azure Resource Manager (auto SP) NTG-Analytics-RG-Dev
north-intelligence-azure-prod Azure Resource Manager (auto SP) NTG-Analytics-RG-Prod

Steps:

  1. Click "New service connection"
  2. Select "Azure Resource Manager"
  3. Choose "Service principal (automatic)"
  4. Select your Azure subscription
  5. Scope to the resource group
  6. Name it as above
  7. Check "Grant access permission to all pipelines"

2. Environments with Approval Gates

Go to: Pipelines → Environments → New environment

Environment Name Approval Required Approvers
north-intelligence-dev No —
north-intelligence-prod Yes Team lead / you

To add approval to PROD:

  1. Open north-intelligence-prod environment
  2. Click ⋮ (three dots) → "Approvals and checks"
  3. Click "+" → "Approvals"
  4. Add required approver(s)
  5. Save

3. Variable Group (Secrets)

Go to: Pipelines → Library → + Variable group

Group name: north-intelligence-secrets

Variable Name Value Secret?
DEV_POSTGRES_CONNECTION_STRING postgresql://user:pass@northintelligence-db-dev:5432/postgres Yes
PROD_POSTGRES_CONNECTION_STRING postgresql://user:pass@northintelligence-db:5432/postgres Yes

Steps:

  1. Create the variable group
  2. Add each variable, toggle the lock icon to mark as secret
  3. Click "Pipeline permissions" → authorize the pipeline

4. Create the Pipeline

Go to: Pipelines → New pipeline

  1. Select "Azure Repos Git"
  2. Select north-intelligence repository
  3. Select "Existing Azure Pipelines YAML file"
  4. Branch: dev (or your feature branch)
  5. Path: /azure-pipelines.yml
  6. Click "Run" to validate

Database Migration Workflow (Alembic)

Creating a new migration

cd backend

# Create a migration file
alembic revision -m "add_new_column_to_orders"

# Edit the generated file in alembic/versions/
# Write SQL in upgrade() and downgrade()

Example migration file

def upgrade() -> None:
    op.execute("""
        ALTER TABLE tracking_3d.shift_logs
        ADD COLUMN notes TEXT;
    """)

def downgrade() -> None:
    op.execute("""
        ALTER TABLE tracking_3d.shift_logs
        DROP COLUMN notes;
    """)

Testing locally

# Apply all pending migrations
alembic upgrade head

# Check current version
alembic current

# Rollback one migration
alembic downgrade -1

How it works in the pipeline

  1. Pipeline extracts the ZIP artifact
  2. Checks if alembic_version table exists in the database
  3. First run: stamps the baseline (marks DB as "at revision 0001")
  4. Subsequent runs: runs alembic upgrade head (applies only new migrations)
  5. Then deploys the app code

Rollback Strategy

App rollback

Re-run a previous successful pipeline build from Azure DevOps:

  1. Go to Pipelines → select the pipeline → Runs
  2. Find the last successful run
  3. Click "Re-run" → deploys the old artifact

DB rollback

# Rollback last migration (run from pipeline agent or locally)
alembic downgrade -1

Emergency: Azure Portal

  1. Go to App Service → Deployment Center
  2. Use "Redeploy" on a previous deployment

Key Design Decisions

Decision Reason
npm ci over npm install Deterministic install from lockfile, faster in CI
CI=false in build step CRA treats warnings as errors when CI=true (Azure DevOps default)
pr: none in pipeline Pipeline triggers on merge (push to branch), not on PR creation
Alembic runs before deploy Ensures DB schema is ready before new code goes live
No secrets in YAML DB connection strings stored in Azure DevOps variable groups (marked secret)
Startup keeps pip install Matches current behavior, simple and reliable
