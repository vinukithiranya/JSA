2. Target Standard

  • Azure DevOps repo, dev → main branch flow, PR-gated both ways.
  • azure-pipelines.yml at repo root: Build & Package (Node 20, npm ci, npm run build:deploy, copy build → backend/static, zip backend/) → Deploy DEV (auto, Python 3.12, alembic upgrade head, ZIP Deploy, health check) → Deploy PROD (manual approval gate, same steps against prod resources).
  • Two fully separate resource groups / App Services / Postgres servers (DEV: B1 + Burstable B1ms, PROD: P1v2 + General Purpose D2s).
  • Secrets live in an Azure DevOps variable group, marked secret — never in YAML or docs.
  • Schema changes go through Alembic migration files, applied by the pipeline — not by app startup code.
  • Every endpoint behind get_current_user/get_user_email; new services check check_service_access.
  • Layered code: Routes (thin) → Repositories (SQL only) → Services (business logic only) → Models (Pydantic validation).
  • DB rules: UUID PKs, TIMESTAMPTZ, created_at/updated_at everywhere, created_by/updated_by on user-editable tables, idempotent migrations, company_id FK on new tables unless documented otherwise.
  • Requirements doc → design doc → implement → verify, for anything beyond a small (<3 file) change.

3.2 Schema migrations

Schema is still created via Base.metadata.create_all() plus a hand-rolled _run_migration() in main.py that runs raw ALTER TABLE statements on every app startup inside a bare try/except: pass — real errors are silently swallowed and there is no versioned history of schema changes.

Fix — step by step
  1. Add Alembic: pip install alembic==1.13.*, add to requirements.txt.
  2. From backend/: alembic init alembic.
  3. In alembic.ini, delete the hardcoded sqlalchemy.url line.
  4. In alembic/env.py, point Alembic at the app's models and env-configured URL: 
    from app.core.config import get_settings
from app.core.db import Base
from app.models import db_models  # noqa: F401

config.set_main_option("sqlalchemy.url", get_settings().database_url)
target_metadata = Base.metadata
  5. Generate the baseline revision and review it against the current schema: 
    alembic revision --autogenerate -m "baseline_current_schema"
  6. For every environment that already has a live database, mark it as already at baseline without re-running the DDL: alembic stamp baseline_current_schema (do not run upgrade for this first revision against existing data).
  7. Convert each of the existing _run_migration() calls into its own versioned migration, guarded for idempotency, e.g.: 
    def upgrade():
    op.execute("ALTER TABLE actions ADD COLUMN IF NOT EXISTS priority VARCHAR(20) DEFAULT 'medium'")

def downgrade():
    op.execute("ALTER TABLE actions DROP COLUMN IF EXISTS priority")
    Repeat for due_date, labels, action_type, linked_issue_id, linked_jsa_id, created_by on actions; event_type, link on notifications; and supervisor_signature on inspection_records.
  8. Delete the _run_migration() function and every call site from main.py's lifespan. Keep Base.metadata.create_all(bind=engine) only for a throwaway local sqlite dev DB, or remove it entirely once Alembic fully owns schema creation.
  9. Test against a fresh empty database: alembic upgrade head should build the entire schema from nothing; run the backend test suite against it.
  10. Add alembic upgrade head as an explicit deploy step (pipeline task, run before the app starts) rather than something the app does at boot.

3. Pipeline definition
Fix — step by step
  1. Commit azure-pipelines.yml at the repo root (adjust task inputs once Open Decision 1 — container vs. ZIP deploy — is settled; the example below assumes ZIP deploy to match the reference architecture): 
    trigger:
  branches:
    include: [dev, main]
pr: none

stages:
- stage: Build
  displayName: 'Build & Package'
  jobs:
  - job: BuildJob
    pool: { vmImage: 'ubuntu-latest' }
    steps:
    - task: NodeTool@0
      inputs: { versionSpec: '20.x' }
    - script: |
        cd frontend
        npm ci
        npm run build:deploy
      displayName: 'Build frontend'
    - task: CopyFiles@2
      inputs:
        sourceFolder: 'frontend/dist'
        targetFolder: 'backend/static'
    - script: test -f backend/static/index.html
      displayName: 'Verify index.html exists'
    - task: ArchiveFiles@2
      inputs:
        rootFolderOrFile: 'backend'
        includeRootFolder: false
        archiveFile: '$(Build.ArtifactStagingDirectory)/backend.zip'
    - publish: '$(Build.ArtifactStagingDirectory)/backend.zip'
      artifact: backend

- stage: DeployDev
  displayName: 'Deploy DEV'
  condition: and(succeeded(), eq(variables['Build.SourceBranch'], 'refs/heads/dev'))
  variables:
  - group: rigpro-secrets
  jobs:
  - deployment: DeployDev
    environment: rigpro-dev
    pool: { vmImage: 'ubuntu-latest' }
    strategy:
      runOnce:
        deploy:
          steps:
          - task: UsePythonVersion@0
            inputs: { versionSpec: '3.12' }
          - script: |
              pip install -r backend/requirements.txt alembic
              cd backend && alembic upgrade head
            env:
              DATABASE_URL: $(DEV_POSTGRES_CONNECTION_STRING)
            displayName: 'Run Alembic migrations'
          - task: AzureWebApp@1
            inputs:
              azureSubscription: 'rigpro-azure-dev'
              appType: webAppLinux
              appName: 'rigpro-app-dev'
              package: '$(Pipeline.Workspace)/backend/backend.zip'
          - script: curl --fail https://rigpro-app-dev.azurewebsites.net/api/health
            displayName: 'Health check'

- stage: DeployProd
  displayName: 'Deploy PROD'
  condition: and(succeeded(), eq(variables['Build.SourceBranch'], 'refs/heads/main'))
  variables:
  - group: rigpro-secrets
  jobs:
  - deployment: DeployProd
    environment: rigpro-prod
    pool: { vmImage: 'ubuntu-latest' }
    strategy:
      runOnce:
        deploy:
          steps:
          - task: UsePythonVersion@0
            inputs: { versionSpec: '3.12' }
          - script: |
              pip install -r backend/requirements.txt alembic
              cd backend && alembic upgrade head
            env:
              DATABASE_URL: $(PROD_POSTGRES_CONNECTION_STRING)
            displayName: 'Run Alembic migrations'
          - task: AzureWebApp@1
            inputs:
              azureSubscription: 'rigpro-azure-prod'
              appType: webAppLinux
              appName: 'rigpro-app'
              package: '$(Pipeline.Workspace)/backend/backend.zip'
          - script: curl --fail https://rigpro-app.azurewebsites.net/api/health
            displayName: 'Health check'

6. Open Decisions (blocking §3.3 and parts of §3.4)

1. Container (current Dockerfile) vs. ZIP Deploy (Northsails' App Service pattern)?

The reference pipeline assumes a non-container App Service with pip install + ZIP deploy. RigPro currently builds a Docker image. Keeping Docker means the pipeline looks more like build-and-push-to-ACR-then-deploy; matching the doc exactly means dropping the Dockerfile-based deploy in favor of ZIP deploy.

2. Azure DevOps vs. GitHub + Azure Pipelines?

Do you have (or want) an northsails.visualstudio.com project for this repo, or should the pipeline run from the existing GitHub repo?

3. Single-tenant vs. multi-company?

Should RigPro adopt company_id on all tables (Northsails' default rule), or is it intentionally single-tenant?

4. UUID primary keys?

Worth the migration effort to switch existing string IDs to UUIDs, or keep current IDs and treat that rule as non-binding for this app?
