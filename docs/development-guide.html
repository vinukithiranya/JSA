Development Guide

May 2026

Workflow

New service or major feature? Follow this order:

  1. Requirements — docs/requirements/[service].html (use template)
  2. Design — docs/design/[service].html (use template)
  3. Implement — Database first, then backend, then frontend
  4. Verify — Test multi-company, access control, end-to-end

Bug fixes and small enhancements (< 3 files) can skip steps 1–2.

Rules

Documentation first

No new service without a requirements doc. No new tables without a design doc.

Multi-company by default

Every new table needs a company_id FK unless there's a documented reason not to.

Layer separation

  • Routes — Thin: auth check + delegate to repo/service
  • Repositories — SQL only, no business logic
  • Services — Business logic only (no FastAPI types, no SQL)
  • Models — Pydantic validation only

Project layout

  • Place new service repositories under backend/repositories/<domain>/ (e.g. repositories/machineshop/machineshop_repository.py) — not at the top level of repositories/.
  • Place new service business logic under backend/services/<domain>/ (e.g. services/machineshop/machineshop_service.py).
  • Cross-service helpers (user, company, group, base) live in repositories/shared/.
  • Frontend components are already grouped by service under frontend/src/components/<service>/ — keep new code in the matching folder.

Security

  • SQL: always %s params, never f-strings
  • Auth: every endpoint uses get_current_user or get_user_email
  • Access: new services check check_service_access
  • No .env or credentials in code

Database

  • UUID primary keys, TIMESTAMPTZ for dates
  • created_at / updated_at on every table
  • created_by / updated_by on user-editable tables
  • Idempotent migrations (IF NOT EXISTS)
  • Update backend/database/schema after every migration

Frontend

  • npm run build:deploy (not npm run build)
  • Material UI only, functional components only
  • Service tabs conditional on accessible_services

Runtime

  • Python 3.12 for backend (CI and App Service)
  • Node.js 20 for frontend build
  • Health endpoint: GET /api/health — DB-ping liveness probe; App Service should be configured to use it
  • Logs go to stdout (configured in backend/main.py) — App Service log stream captures them

Follow existing patterns

The 3D Tracking service is the reference implementation. When in doubt, look at how it does it.

Git

  • Develop on dev, feature branches for large changes
  • Never commit .env, node_modules/, venv/, __pycache__/

Verification Checklist

Before deploying:

  • API endpoints return correct data
  • Multi-company filtering works
  • Access control enforced (403 for unauthorized)
  • Frontend renders correctly
  • Migration is idempotent
  • Schema file updated
  • npm run build:deploy succeeds
  • No hardcoded values or debug code
