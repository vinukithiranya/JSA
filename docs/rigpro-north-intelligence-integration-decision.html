RigPro × North Intelligence — Standalone vs. Integrate

July 2026 Recommendation: standalone + SSO/nav integration

Prompted by: manager's suggestion to integrate RigPro with the North Intelligence platform to lower development and maintenance cost. This assessment is based on a direct audit of the North Intelligence codebase at C:\Users\vinukiT\Downloads\north-intelligence — not just its published docs.

Recommendation

Keep RigPro as its own standalone app (own FastAPI process, own Postgres database, own SQLAlchemy models, own Vite frontend). Integrate only at the login and navigation level: shared Azure AD / SSO so users log in once, plus a tile/link inside North Intelligence's home page pointing at RigPro's own URL.

This gets the "one place our team goes, one login" outcome non-technical stakeholders usually mean by "integrate," without forcing either codebase to violate its own architecture.

Why Full Integration Doesn't Actually Lower Cost Here

North Intelligence is a single-process monorepo (one FastAPI app, one Postgres instance, one React SPA, one deploy pipeline) built on two architectural decisions that directly conflict with how RigPro is built today:

1
No ORM, by formal decision

North Intelligence has a reaffirmed ADR (adr-001-raw-sql-over-orm.html) banning SQLAlchemy-as-ORM — everything is raw SQL via psycopg2 + RealDictCursor. RigPro is built entirely on SQLAlchemy 2.0 ORM models, and its repositories/ layer was just built around that ORM. Integrating as a real service means either rewriting all 17 domains off the ORM, or permanently running a second, contradictory data-access pattern inside their monorepo.

2
Different frontend stack

North Intelligence is Create React App + MUI v7 throughout. RigPro is Vite + React 18 with no MUI. Merging means rebuilding RigPro's UI on CRA and likely re-skinning it in MUI to look native inside the shell.

3
Different auth model

North Intelligence trusts Azure Easy Auth headers (X-MS-CLIENT-PRINCIPAL-NAME) injected by the App Service platform — no JWT validation in app code at all. RigPro's newly-built JWT auth (core/security.py, get_current_user) would become dead code in a true integration; access would instead be gated through their azure_groups(company_id, service_id, role) model.

4
No plugin/registry mechanism

Adding a "service" inside North Intelligence means manually touching ~7 places: a router file, a main.py include_router call, a ServiceDefinition dict entry in config/services.py, a frontend route, a hardcoded array entry in HomePage.js, a new component folder, and a new migration folder. Even their own reference service ("3D Tracking") doesn't consistently follow their documented layering — there's no framework doing this for you.

5
Shared blast radius

One FastAPI process and one deploy pipeline means a bug or bad migration in RigPro could break North Intelligence's other 12 services' deploy, and vice versa — where today they fail independently.

RigPro is a full production application (~35 users, 17 domains) — not a small feature. Full integration would front-load a large rewrite (ORM → raw SQL, Vite → CRA, JWT → Easy-Auth-headers) before any maintenance savings could begin, and would require taking on North Intelligence's own architectural debt (partial Alembic adoption, inconsistent service layering) as RigPro's problem too.

The Lightweight Integration Path (Recommended)

  1. Put RigPro behind the same Azure AD tenant / SSO that North Intelligence uses, so users authenticate once.
  2. Add a tile or link for RigPro inside North Intelligence's HomePage.js services list, pointing at RigPro's own URL (optionally reverse-proxied under the same domain for a seamless feel).
  3. Keep RigPro's own FastAPI process, Postgres database, SQLAlchemy models, and Vite frontend exactly as they are.

This captures the "single pane of glass, one login" value without either app needing to abandon its own architecture, and without merging two different data-access and UI conventions into one codebase long-term.

When Full Monorepo Integration Would Actually Make Sense

  • If RigPro were small — a handful of screens, not 17 domains — the rewrite cost would be proportionally low.
  • If RigPro were being built from scratch today and could adopt raw-SQL + CRA + Easy-Auth from day one, instead of retrofitting an existing app onto someone else's conventions.

Neither condition applies today.
